实验报告

1.实验目的

  1. 安装并运行一个桌面网络软件（Cisco Packet Tracer）；理解C/S、B/S、P2P等不同网络软件类型与架构； 通过动手搭建一个简单的局域网，直观感受网络软件通信的基本原理。
  2. 安装并运行一个Web博客软件（本实验选用 Hexo 静态博客框架）；了解和掌握Web开发环境的基础安装与配置（Node.js、npm、Git）；理解Web软件的B/S架构特点以及静态站点与动态站点的区别。
  3. 安装Visual Studio Code（VS Code）文本代码编辑器；安装和配置Markdown相关插件，掌握代码编辑器插件的基本概念；学习Markdown基础语法，使用VS Code完成Markdown文档的编辑和渲染任务。
  4. 使用静态站点构建工具（Hexo）构建一个静态Web博客站点；了解和掌握Web应用开发的基本要素与流程：环境准备 → 项目初始化 → 内容编写 → 主题配置 →本地预览 → 部署上线；理解静态站点生成器（SSG）的工作原理及其与动态站点的区别。
  5. 使用虚拟机工具（VirtualBox）安装Linux系统（Ubuntu Server）；通过克隆技术构建一个多机虚拟化环境；了解和掌握基于虚拟机的开发环境的基本构建方法和配置。
  6. 使用命令行进行Linux系统的基础操作；了解和简单管理常用系统配置文件（ /etc/passwd 、 /etc/group 、 /etc/profile 等）；了解和简单管理用户配置文件（ ~/.bashrc 、 ~/.profile 等）；掌握环境变量的查看、设置与管理方法。
  7. 安装和配置OpenSSH服务，实现多机之间的远程登录管理；配置SSH密钥认证，实现免密登录；了解多机开发环境下的账户管理方法；掌握使用SCP和rsync进行跨机文件传输与同步。
  8. 综合运用前七个实验所学的各类工具；构建和模拟一个完整的网络环境；完整体验和实现Web静态站点从开发到部署的全过程；掌握自动化部署（CI/CD）的基本概念与实践方法。

2.实验环境

  • 操作系统：Windows 11
  • 软件版本：Cisco Packet Tracer、Node.js（LTS版本，含npm）、Git for Windows、Hexo CLI、Visual Studio Code、Oracle VirtualBox、Ubuntu Server 22.04 LTS ISO镜像
  • 虚拟机配置：VirtualBox 7.0，Ubuntu Server 22.04 LTS，内存2GB，硬盘25GB
  • 网络拓扑：Host-Only网络，IP段192.168.56.0/24

实验一：网络软件基础

步骤1：下载安装Cisco Packet Tracer

打开浏览器，访问Cisco Networking Academy官网点击页面右上角注册账号，登录后，在课程或资源页面中找到 “Packet Tracer” 下载入口。选择适合你操作系统的版本（如 PacketTracer821.exe for Windows），点击下载。找到下载好的安装文件，右键点击，选择 “以管理员身份运行” 。根据安装向导欢迎界面引导， 完成安装。

步骤2：搭建第一个局域网

  1. 添加设备：在左侧设备工具栏中，找到 “Network Devices” 下的 “Switches” （交换机图标）。将 “2960” 型号的交换机拖拽到中央工作区。再找到 “End Devices”，将两台PC-PT拖拽到工作区。

  2. 连接设备：点击设备工具栏中的 “Connections” （连线工具，图标为闪电或线缆）。选择 “Copper Straight-Through” （直通线）。依次点击 PC1 → 选择 FastEthernet0 端口 → 再点击 交换机 → 选择某个端口（如FastEthernet0/1 ）。用同样的方法，将 PC2 连接到交换机的另一个端口（如 FastEthernet0/2 ）。

    *图2：连接设备

  3. 配置IP地址并测试网络连通性：双击PC1，在弹出的窗口中选择 “Desktop” 选项卡。点击 “IP Configuration” 。在IP Address栏中输入 192.168.1.1 ，Subnet Mask自动填充为 255.255.255.0 。关闭PC1的配置窗口。双击PC2，同样进入 Desktop → IP Configuration。将IP Address设为 192.168.1.2 ，子网掩码保持 255.255.255.0 。

知识总结

1. 三种主流网络软件架构对比

架构类型 全称 核心特征 典型例子 优缺点
C/S 客户端/服务器 需安装专用客户端，客户与服务器分工协作 微信、网络游戏、Outlook 安全性高、性能好；但维护成本高
B/S 浏览器/服务器 仅需浏览器即可访问，核心功能在服务端 淘宝、知乎、在线办公软件 跨平台、免安装；依赖网络和服务器性能
P2P 点对点 每个节点既是客户端也是服务器，可直接通信 BT下载、部分局域网通讯工具 去中心化、资源利用率高；管理难度大

2. 静态博客 vs 动态博客

类型 原理 特点 典型代表
静态博客 本地将Markdown预编译为纯HTML 加载快、安全性高、无需数据库 Hexo、Hugo、VuePress
动态博客 每次访问由服务器实时生成页面 功能丰富、有后台管理、需要数据库 WordPress、Typecho

3. Web开发环境核心组件

组件 作用 本实验对应工具
运行时环境 执行编程语言代码 Node.js（运行JavaScript）
包管理器 管理项目依赖库 npm（Node Package Manager）
版本控制 代码管理与协作 Git
构建工具 将源文件转换为可部署网站 Hexo CLI

4. 代码编辑器 vs 集成开发环境（IDE）

类型 特点 典型代表
代码编辑器 轻量级、启动快、以编辑为核心、通过插件扩展 VS Code、Sublime Text
集成开发环境（IDE） 功能全面、集成编译/调试/构建全套工具链 IntelliJ IDEA、PyCharm

5. 核心插件功能归纳

插件名称 核心功能
Markdown All in One 自动生成目录（[TOC]）、快捷键支持、列表自动延续、键盘快捷操作
Markdown Preview Enhanced 实时渲染预览、支持LaTeX数学公式、Mermaid流程图、导出PDF/HTML
Paste Image 直接粘贴剪贴板图片到Markdown，并自动保存至相对路径
markdownlint 检查Markdown语法错误，规范写作格式

6. 静态站点生成器（SSG）核心原理

对比维度 静态站点（SSG） 动态站点（如WordPress）
页面生成时机 构建时预先生成 用户访问时实时生成
数据库 不需要 通常需要
访问速度 极快（纯静态文件） 较慢（需执行后端代码+查库）
部署方式 上传静态文件即可 需配置服务器环境和数据库

7. 虚拟机网络的三种核心模式

网络模式 特点 典型用途
NAT 虚拟机通过宿主机上网，外部网络不可直达内部 日常上网、安装软件
桥接（Bridged） 虚拟机直接接入宿主机所在局域网，拥有独立IP 需要被局域网内其他设备直接访问
仅主机（Host-Only） 宿主机与虚拟机形成隔离的私有网络 搭建内网互通实验环境（本实验多机互联）

8. 关键系统配置文件

文件路径 作用 加载时机
/etc/passwd 用户账户信息（用户名、UID、GID、家目录、Shell） 系统启动
/etc/shadow 用户加密后的密码（仅root可读） 系统启动
/etc/group 用户组信息 系统启动
/etc/profile 系统级全局变量（所有用户） 用户登录时
/etc/environment 系统级环境变量（全用户，不依赖Shell） 系统启动

9. 用户级配置文件

文件路径 作用 加载时机
~/.bashrc 定义用户别名、函数、私有环境变量 每次打开新终端（Bash）
~/.profile 用户登录时执行一次的环境设置 用户登录时

10.文件权限数字法

数字 权限 含义
7 rwx 读 + 写 + 执行
6 rw- 读 + 写
5 r-x 读 + 执行
4 r– 只读
0 — 无权限

11. SSH 两种认证方式对比

认证方式 原理 安全性 适用场景
密码认证 每次登录输入用户密码 较低（易被暴力破解） 临时访问
密钥认证 使用公钥/私钥非对称加密验证身份 高（可禁用密码登录） 日常管理、自动化脚本（免密）

12. SCP vs Rsync 对比

工具 特点 适用场景
SCP 基于SSH的简单复制，每次全量传输 一次性、少量文件传输
Rsync 增量同步，只传输变化的部分，支持压缩（-z）、归档（-a） 目录同步、定期备份、大规模文件同步

Rsync 常用参数

  • -a：归档模式（保留权限、时间戳）
  • -v：显示详细过程
  • -z：传输时压缩
  • -e ssh：指定使用SSH协议
  • --delete：删除目标中源已不存在的文件（保持完全一致）

13. CI/CD 核心理念与本实验映射

概念 含义 本实验体现
持续集成（CI） 代码变更后自动进行构建和测试 推送代码至GitHub后，自动触发 hexo generate
持续部署（CD） 构建通过后自动部署到生产环境 生成的静态文件自动推送到 GitHub Pages 或云服务器

14. 自动化部署 Vs 手动部署

方式 命令/触发条件 适用场景
手动部署 本地 hexo g + rsync 推送到服务器 调试阶段、小型个人项目
自动化部署（CI/CD） git push 触发GitHub Actions 团队协作、生产环境、版本管理规范的项目

出现问题

  1. 虚拟机ping测试时可以ping通其他虚拟机，但是无法ping通宿主机。
    解决方法：
    先检查了虚拟机的网卡，网卡1为NAT，网卡2为Host-only,没有问题，再在宿主机作测试，发现是宿主机防火墙的问题，关闭宿主机防火墙后成功虚拟机成功ping通宿主机。
  2. 使用rsync同步文件时报错not recognized as the name of a cmdlet
    解决方法：安装Chocolatey包管理器后用choco install rsync下载rsync

心得体会

通过本次实验，我深刻体会到了工具链的概念。工具链不是工具的简单堆叠，而是一条相互衔接的工作流水线，就像本实验中的VS Code、Hexo、Git、Github Action等，我用VS Code写Markdown，用Hexo生成静态页面，用Git推送代码，用GitHub Actions自动部署，实现了博客的自动推送部署。在这个过程中，每一个环节都有其不可替代的作用，每个工具都在发挥它自己的作用，缺少一个都会导致流程中断。而要让这些工具发挥出作用，需要我动手实践，在实践中学会这些工具的应用，只有深刻了解了这些工具的原理，才能在出现错误时拥有排错的能力。
在学习使用这些工具的同时，我也注意到国内虽然也有Codeup、Gitee等平台，但在生态建设、知名度上都不如GitHub。这恰恰是投身其中、贡献力量的契机：以参与开源为起点，助力国产软件生态的繁茂生长。
